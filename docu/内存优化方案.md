# AI Interview 内存优化方案

> 针对 2C2G 服务器的资源优化技术文档

---

## 背景

**服务器配置**：2 核 CPU，2GB 内存

**挑战**：
- 需要运行多个服务（PostgreSQL、Redis、RabbitMQ、Go 应用）
- 原架构包含 Milvus 向量数据库（内存占用 1GB+）
- 需要支持并发面试场景（目标：100 并发）

**目标**：
- 总内存占用 < 2GB
- 保证核心功能正常运行
- 避免 OOM（Out of Memory）导致服务崩溃

---

## 优化方案

### 1. 向量检索优化

#### 问题分析

**原方案**：Milvus 向量数据库
- 最小内存需求：1GB+
- 运行时开销：索引加载、查询缓存
- 2GB 内存无法运行

**题库规模**：
- 5000 题
- 每题 1536 维 embedding（OpenAI text-embedding-3-small）
- 向量数据大小：`5000 * 1536 * 4 bytes ≈ 30MB`

#### 解决方案：PostgreSQL + pgvector

**技术选型**：
- 使用 PostgreSQL 的 pgvector 扩展
- 内存占用：与 PostgreSQL 共享，增加约 100MB
- 查询性能：5000 题规模下，50-200ms（可接受）

**实现细节**：

```sql
-- 1. 安装 pgvector 扩展
CREATE EXTENSION vector;

-- 2. 创建题库表
CREATE TABLE interview_questions (
    id UUID PRIMARY KEY,
    question_text TEXT NOT NULL,
    embedding vector(1536),  -- 1536 维向量
    tags TEXT[],             -- 技术标签
    difficulty VARCHAR(16),  -- easy/medium/hard
    standard_answer TEXT,
    follow_up_hints JSONB,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 3. 创建向量索引（IVFFlat）
CREATE INDEX ON interview_questions 
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);

-- 4. 向量检索查询
SELECT id, question_text, 
       1 - (embedding <=> $1::vector) AS similarity
FROM interview_questions
WHERE tags && $2::text[]  -- 标签过滤
ORDER BY embedding <=> $1::vector
LIMIT 20;
```

**Docker Compose 配置**：

```yaml
services:
  postgres:
    image: pgvector/pgvector:pg16
    environment:
      POSTGRES_SHARED_BUFFERS: 256MB
      POSTGRES_EFFECTIVE_CACHE_SIZE: 512MB
      POSTGRES_WORK_MEM: 16MB
    deploy:
      resources:
        limits:
          memory: 400M
        reservations:
          memory: 300M
```

**性能对比**：

| 方案 | 内存占用 | 查询延迟 | 成本 |
|------|---------|---------|------|
| Milvus | 1GB+ | 10-50ms | 高 |
| pgvector | 100MB | 50-200ms | 低 |
| 标签匹配 | 0 | < 10ms | 最低 |

**结论**：pgvector 是 2C2G 场景的最佳选择

### 2. 对话历史优化

#### 问题分析

**内存占用估算**：
- 单场面试：20 轮对话，每轮 500 字 = 10KB
- 100 场并发：100 * 10KB = 1MB（可接受）
- 风险：长对话（50+ 轮）+ 无 TTL = 内存泄漏

#### 解决方案：TTL + 裁剪 + 分层存储

**策略 1：严格的 TTL**

```go
// Redis 存储对话历史，2 小时后自动过期
func (s *Service) SaveHistory(ctx context.Context, interviewID string, history []Message) error {
    key := fmt.Sprintf("interview:%s:history", interviewID)
    data, _ := json.Marshal(history)
    return s.redis.Set(ctx, key, data, 2*time.Hour).Err()
}
```

**策略 2：对话裁剪**

```go
// 保留 system prompt + 最近 15 轮对话
func (s *Service) AppendHistory(ctx context.Context, interviewID string, msg Message) error {
    history := s.GetHistory(ctx, interviewID)
    history = append(history, msg)
    
    // 裁剪：超过 16 条（1 system + 15 轮），只保留最新的
    if len(history) > 16 {
        systemPrompt := history[0]
        recentMessages := history[len(history)-15:]
        history = append([]Message{systemPrompt}, recentMessages...)
    }
    
    return s.SaveHistory(ctx, interviewID, history)
}
```

**策略 3：分层存储**

| 存储层 | 用途 | TTL | 说明 |
|--------|------|-----|------|
| Redis | 工作集（当前面试） | 2 小时 | 高性能读写 |
| PostgreSQL | 完整历史 | 永久 | 用于报告生成 |

```go
// 每轮对话同时写入 PostgreSQL
func (s *RecordService) SaveTurn(ctx context.Context, turn *domain.InterviewTurn) error {
    _, err := s.db.Exec(ctx,
        "INSERT INTO interview_turns (interview_id, turn_id, question, user_answer, asr_raw) VALUES ($1, $2, $3, $4, $5)",
        turn.InterviewID, turn.TurnID, turn.Question, turn.UserAnswer, turn.ASRRaw,
    )
    return err
}
```

**内存节省**：约 50MB（避免历史堆积）

### 3. 并发控制优化

#### 问题分析

**风险点**：
- LLM 调用：每个请求 2-5 秒，大量并发会堆积 goroutine
- 文件上传：5MB PDF 在内存中处理，10 并发 = 50MB+
- HTTP 连接池：无限制会耗尽文件描述符

#### 解决方案：信号量限流

**LLM 并发限制**：

```go
// 全局信号量，限制最多 20 个并发 LLM 调用
var llmSemaphore = make(chan struct{}, 20)

func (n *LLMNode) Invoke(ctx context.Context, input string) (string, error) {
    // 获取令牌
    select {
    case llmSemaphore <- struct{}{}:
        defer func() { <-llmSemaphore }()
    case <-ctx.Done():
        return "", errors.New("context canceled")
    }
    
    // 调用 LLM
    ctx, cancel := context.WithTimeout(ctx, 10*time.Second)
    defer cancel()
    
    return n.client.Generate(ctx, input)
}
```

**文件上传并发限制**：

```go
// 限制最多 5 个并发上传
var uploadSemaphore = make(chan struct{}, 5)

func (h *Handler) UploadResume(ctx context.Context, c *app.RequestContext) {
    select {
    case uploadSemaphore <- struct{}{}:
        defer func() { <-uploadSemaphore }()
    default:
        c.JSON(429, map[string]interface{}{
            "code":    1003,
            "message": "上传繁忙，请稍后重试",
        })
        return
    }
    
    // 处理上传
}
```

**HTTP 客户端配置**：

```go
// LLM 客户端连接池
httpClient := &http.Client{
    Transport: &http.Transport{
        MaxIdleConns:        50,
        MaxIdleConnsPerHost: 10,
        IdleConnTimeout:     90 * time.Second,
        DisableKeepAlives:   false,
    },
    Timeout: 30 * time.Second,
}
```

### 4. 文件处理优化

#### 问题分析

**内存占用**：
- 用户上传 5MB PDF
- 读取到内存：5MB
- PDF 解析库临时内存：10MB+
- 10 并发上传：150MB+

#### 解决方案：流式处理 + 大小限制

**策略 1：文件大小限制**

```go
// Hertz 中间件
func FileSizeLimitMiddleware(maxSizeMB int64) app.HandlerFunc {
    maxSize := maxSizeMB * 1024 * 1024
    return func(ctx context.Context, c *app.RequestContext) {
        if c.Request.Header.ContentLength() > maxSize {
            c.JSON(400, map[string]interface{}{
                "code":    1001,
                "message": fmt.Sprintf("文件大小超过限制（最大 %dMB）", maxSizeMB),
            })
            c.Abort()
            return
        }
        c.Next(ctx)
    }
}
```

**策略 2：流式处理**

```go
// 从 S3 流式读取，避免全部加载到内存
func (s *Service) ParseResume(ctx context.Context, s3Key string) (*domain.StructuredResume, error) {
    // 1. 流式读取 S3 对象
    reader, err := s.s3Client.GetObject(ctx, s3Key)
    if err != nil {
        return nil, err
    }
    defer reader.Close()
    
    // 2. 流式解析 PDF（使用 io.Reader，不全部加载）
    text, err := pdf.ExtractTextStreaming(reader)
    if err != nil {
        return nil, err
    }
    
    // 3. LLM 提取
    return s.llmExtractor.Extract(ctx, text)
}
```

**策略 3：S3 预签名 URL 上传**

```go
// 前端直接上传到 S3，后端不经手文件内容
func (h *Handler) GetUploadURL(ctx context.Context, c *app.RequestContext) {
    userID := c.GetString("user_id")
    s3Key := fmt.Sprintf("resumes/%s/original.pdf", userID)
    
    // 生成预签名 URL（5 分钟有效）
    url, err := h.s3Client.PresignPutObject(ctx, s3Key, 5*time.Minute)
    
    c.JSON(200, map[string]interface{}{
        "code": 0,
        "data": map[string]string{
            "upload_url": url,
            "s3_key":     s3Key,
        },
    })
}
```

**内存节省**：约 100MB（避免大文件在内存中处理）

### 5. 日志优化

#### 问题分析

**风险点**：
- Debug 日志：每个请求 10+ 条日志
- 100 QPS：1000 条日志/秒
- 日志写磁盘：IO 瓶颈 + 磁盘占用

#### 解决方案：级别控制 + 采样 + 轮转

**策略 1：生产环境用 Info 级别**


```go
    if err == nil && rand.Float64() < 0.01 {
        slog.Debug("cache hit", "key", key)
    }
    
    return val, err
}
```

**策略 3：避免记录大对象**

```go
// ❌ 错误：记录完整 prompt
```

### 7. 简历存储优化

#### 问题分析

**原方案**：Redis 主存储
- 单份简历：5KB
- 40 万份简历：2GB（Redis 会爆）

#### 解决方案：PostgreSQL 主 + Redis 缓存

**存储策略**：

| 存储层 | 用途 | 容量 | 性能 |
|--------|------|------|------|
| PostgreSQL | 主存储（所有简历） | 无限 | 10-50ms |
| Redis | 热缓存（正在面试） | 5MB | < 1ms |

**实现**：

```go
// 提交简历：存 PostgreSQL
func (s *Service) SubmitResume(ctx context.Context, resume *domain.StructuredResume) error {
    data, _ := json.Marshal(resume)
    _, err := s.db.Exec(ctx,
        "INSERT INTO resumes (user_id, data) VALUES ($1, $2) ON CONFLICT (user_id) DO UPDATE SET data = $2",
        resume.UserID, data,
    )
    return err
}

// 获取简历：先查 Redis，miss 则查 PG 并回填
func (s *Service) GetResume(ctx context.Context, userID string) (*domain.StructuredResume, error) {
    key := fmt.Sprintf("resume:%s", userID)
    
    // 1. 尝试 Redis
    data, err := s.redis.Get(ctx, key).Result()
    if err == redis.Nil {
        // 2. Redis miss，查 PostgreSQL
        err = s.db.QueryRow(ctx, "SELECT data FROM resumes WHERE user_id = $1", userID).Scan(&data)
        if err != nil {
            return nil, err
        }
        
        // 3. 回填 Redis，TTL 1 小时
        s.redis.Set(ctx, key, data, 1*time.Hour)
    }
    
    var resume domain.StructuredResume
    json.Unmarshal([]byte(data), &resume)
    return &resume, nil
}
```

**内存节省**：Redis 只存热数据（约 5MB），而不是全部简历（2GB）

## 总结

### 优化效果

| 项目 | 优化前 | 优化后 | 节省 |
|------|--------|--------|------|
| Milvus | 1GB+ | 0（改用 pgvector） | **1GB** |
| 对话历史 | 无限增长 | TTL + 裁剪 | **50MB** |
| 文件处理 | 内存加载 | 流式处理 | **100MB** |
| 简历存储 | Redis 全量 | PG 主 + Redis 缓存 | **1.9GB** |
| **总内存** | **3GB+** | **< 2GB** | **✅ 可运行** |

### 关键技术

1. **pgvector 替代 Milvus**：节省 1GB 内存
2. **分层存储**：热数据 Redis，冷数据 PostgreSQL
3. **并发控制**：信号量限流，避免资源耗尽
4. **流式处理**：避免大文件全部加载到内存
5. **严格 TTL**：防止内存泄漏

### 适用场景

- ✅ 2C2G 小型服务器
- ✅ 初创项目，成本敏感
- ✅ 并发量 < 100
- ❌ 高并发场景（需要扩容）
- ❌ 大规模题库（> 10万题，需要专业向量数据库）
