## 前端层

### 1. WebRTC / MediaDevices

**功能**
- 采集用户麦克风音频流
- 实时将音频流传输至后端 ASR 节点
- 管理音视频设备权限

**技术路线**
- 使用浏览器原生 `getUserMedia` API 采集音频
- 音频流通过 WebRTC 或 AudioWorklet 分片传输
- 建议采用 VAD（语音活动检测）判断用户说话结束，避免持续传输静音数据

---

### 2. SSE / Stream Client

**功能**
- 接收服务端流式推送的 AI 面试官文字回复
- 接收 TTS 音频流并实时播放
- 维护 SSE 长连接，处理断线重连

**技术路线**
- 使用浏览器原生 `EventSource` API 建立 SSE 连接
- 断线重连时携带 `interview_id` 恢复面试状态
- TTS 音频流边接收边播放，使用 `Web Audio API` 实现流式播放

---

### 3. 代码编辑器

**功能**
- 算法阶段提供代码编辑环境
- 支持语法高亮、自动补全
- 用户点击提交后将代码通过 HTTP POST 上传

**技术路线**
- 使用 Monaco Editor（VS Code 同款编辑器）
- 支持主流语言（Java / Python / Go / C++）
- 提交时将代码文本 + 当前 `interview_id` + `question_id` 一并上传

---

### 4. UI Components

**功能**
- 面试全流程页面：上传简历、等待页、面试间、问卷页、报告页
- 面试进度展示（当前阶段、题目数量）
- 问卷填写与提交

**技术路线**
- React + 组件库（待定）
- 面试间布局：左侧对话记录、右侧代码编辑器（算法阶段显示）
- 问卷页每轮对话展示 ASR 文本，用户对每轮打标（good/bad）

## 网关层

### Hertz Server

**功能**
- 统一接入层，处理所有来自前端的请求
- SSE 长连接管理
- 鉴权与中间件处理
- 请求路由分发至各业务服务

**技术路线**
- 使用字节跳动开源 Hertz 框架（Go）
- HAuth 中间件处理 JWT 鉴权
- Request Dispatcher 根据请求类型通过 Kitex RPC 转发至对应业务服务

## 业务服务层

### 1. Interview Service

**功能**
- 面试全生命周期管理（创建、推进、结束）
- 维护面试阶段状态
- 协调各下游服务
- 面试结束后向 MQ 发送消息触发异步报告生成

**技术路线**
- Kitex RPC 服务（Go）
- 调用 Eino Core 进行 Graph Orchestration
- 面试状态存储于 Redis，key 为 `interview:{interview_id}:state`

**面试状态结构**
```json
{
  "interview_id": "xxx",
  "stage": "questioning",
  "questions_asked": 3,
  "current_question_followups": 1,
  "started_at": "2024-01-01T10:00:00Z"
}
```

---

### 2. User / Resume Service

**功能**
- 用户注册、登录、信息管理
- 简历文件上传至 S3
- 触发信息提取 Agent 异步解析简历
- 返回结构化简历数据

**技术路线**
- Kitex RPC 服务（Go）
- 简历原文件存 S3，解析结果存 Redis（key: `resume:{user_id}`）
- 简历解析在用户上传后立即异步触发，确保进入面试时已 ready

**结构化简历 JSON**
```json
{
  "user_id": "xxx",
  "skills": ["Java", "Redis", "MySQL"],
  "projects": [
    {
      "name": "电商平台",
      "tech_stack": ["SpringBoot", "Redis"],
      "description": "...",
      "highlights": ["QPS提升3倍"]
    }
  ],
  "internships": [],
  "education": {
    "school": "xxx大学",
    "major": "计算机科学",
    "graduation": "2025-06"
  }
}
```

---

### 3. Record / Storage Service

**功能**
- 存储每轮对话的 ASR 转写文本
- 存储音频原文件至 S3（合规存档）
- 供 Report Worker 读取完整对话记录
- 供问卷系统关联每轮对话

**技术路线**
- Kitex RPC 服务（Go）
- ASR 文本存 PostgreSQL，结构按轮次存储
- 音频文件存 S3

**对话记录结构**
```json
{
  "interview_id": "xxx",
  "turn_id": "001",
  "stage": "questioning",
  "question": "请介绍一下HashMap的扩容机制",
  "user_answer": "用户的回答文本...",
  "asr_raw": "原始ASR转写文本...",
  "timestamp": "2024-01-01T10:05:00Z"
}
```

---

### 4. Report Worker

**功能**
- 消费 MQ 消息，异步生成面试评分报告
- 从 Record Service 读取完整对话记录
- 调用评分 Agent 生成多维度评分
- 结果存 PostgreSQL，通知前端

**技术路线**
- 独立 Worker 服务（Go）
- 消费 MQ 消息后拉取完整 ASR 文本
- 调用 LLM 生成评分报告

**评分报告结构**
```json
{
  "interview_id": "xxx",
  "dimensions": {
    "knowledge_depth": 8,
    "expression": 7,
    "problem_solving": 9,
    "code_quality": 8,
    "stress_response": 6
  },
  "summary": "候选人基础扎实...",
  "weak_points": ["系统设计深度不足", "边界条件考虑不周"],
  "strong_points": ["算法思路清晰", "表达流畅"]
}
```

## AI层 - Eino Core

整体采用 Graph Orchestration，节点间通过结构化数据传递（A2A形式）。

### 1. ASR Node（语音转文字）

**功能**
- 接收前端音频流，实时转写为文字
- 流式输出，边说边识别
- 检测用户停止说话，输出完整句子给 Router Node

**技术路线**
- 接入流式 ASR 服务（火山引擎 / 阿里云 ASR）
- VAD 判断说话结束
- 输出结果直接进入 Router Node

---

### 2. Router Node（状态机）

**功能**
- 结合当前面试状态 + ASR 文本，决定下一步走哪个分支
- 控制阶段切换（自我介绍→提问→算法→反问）
- 控制题目状态机（出题→追问→关闭→下一题）

**技术路线**
- 纯代码逻辑，不调用 LLM，保证低延迟
- 从 Redis 读取当前面试状态
- 判断规则示例：

```
if stage == "intro" && 用户停止说话:
    → 触发信息提取Agent，切换到提问阶段

if stage == "questioning" && followups >= 2:
    → 关闭当前题，RAG检索下一题

if stage == "questioning" && questions_asked >= N:
    → 切换到算法阶段

if 收到代码提交事件:
    → 触发Code Judge Agent
```

---

### 3. 信息提取 Agent

**触发时机**
- 用户上传简历后（异步解析简历）
- 自我介绍结束后（补充口述信息）

**功能**
- 简历原文 → 结构化 JSON（技术栈、项目、实习经历）
- 自我介绍 ASR 文本 → 提取简历未提及的额外信息
- 结果存入 Redis，供 Interview Agent 全程使用

**技术路线**
- 调用 LLM，prompt 指定输出结构化 JSON
- 结果 merge 进 Redis 上下文

---

### 4. RAG 检索（题库检索）

**功能**
- 根据用户简历技术栈，从八股题库检索相关题目
- 返回候选题目列表，注入 Interview Agent 上下文
- 只在"准备出新题"时触发，不在每轮对话都检索

**技术路线**
- VectorDB 存储题库（Milvus / Qdrant）
- 题目 embedding 基于技术标签检索
- 题库数据结构：

```json
{
  "question_id": "xxx",
  "question": "讲一下HashMap的扩容机制",
  "tags": ["Java", "集合", "中等"],
  "standard_answer": "...",
  "follow_up_hints": [
    "线程安全怎么处理",
    "和ConcurrentHashMap区别"
  ]
}
```

---

### 5. Interview Agent

**功能**
- 提问阶段：根据题库候选题目和用户上下文出题、追问
- 反问阶段：切换角色扮演 prompt，以面试官身份回答用户问题
- 接收 Code Judge Agent 的结构化结果，决定追问方向

**技术路线**
- 调用 LLM Node（Doubao / GPT-4o）
- 维护对话 History，存 Redis，每次请求携带完整 history（无状态设计）
- 提问阶段 system prompt 和反问阶段 system prompt 独立设计
- follow_up_hints 作为追问方向参考注入 prompt

---

### 6. Code Judge Agent

**功能**
- 接收用户提交的代码文本
- 评估代码正确性、时间复杂度、空间复杂度、边界条件处理
- 返回结构化评估结果给 Interview Agent

**技术路线**
- 调用 LLM，prompt 指定输出结构化 JSON
- 题目范围限定在 LeetCode Hot 100，LLM 评估准确性有保证

**返回结构**
```json
{
  "correctness": true,
  "time_complexity": "O(n)",
  "space_complexity": "O(1)",
  "issues": ["边界条件未处理", "变量命名不规范"]
}
```

---

### 7. LLM Node

**功能**
- 统一的 LLM 调用节点
- 流式输出，第一个 token 产出即送往 TTS Node

**技术路线**
- 主选 Doubao，备选 GPT-4o
- 开启 streaming 模式
- 输出第一句话时立即触发 TTS，实现 LLM → TTS 流水线并行

---

### 8. TTS Node（文字转语音）

**功能**
- 接收 LLM 流式文字输出，实时合成语音
- 流式推送音频至前端，边合成边播放

**技术路线**
- 接入流式 TTS 服务（火山引擎 / 阿里云 TTS）
- 通过 SSE 将音频流推送至前端
- 前端使用 Web Audio API 流式播放

## 存储层

### PostgreSQL

| 表 | 存储内容 |
|---|---|
| users | 用户账号信息 |
| interviews | 面试元数据（开始/结束时间、阶段完成情况） |
| interview_turns | 每轮对话 ASR 文本记录 |
| reports | 评分报告结果 |
| questionnaire_results | 问卷标注数据（含 quality 标签） |

---

### Redis

| Key 格式 | 存储内容 | 过期时间 |
|---|---|---|
| `interview:{id}:state` | 面试阶段状态 | 面试结束后清除 |
| `interview:{id}:history` | 对话 History | 面试结束后清除 |
| `resume:{user_id}` | 结构化简历 JSON | 7天 |

---

### S3 / Object Storage

| 路径 | 存储内容 |
|---|---|
| `/resumes/{user_id}/` | 简历原文件（PDF） |
| `/audio/{interview_id}/` | 面试音频录音 |
| `/sft/{date}/` | 导出的 JSONL 训练数据 |

## 消息队列

### MQ Broker

**功能**
- 面试结束后，Interview Service 发布消息
- Report Worker 消费消息，异步生成评分报告
- 解耦面试主流程与报告生成，避免用户等待

**消息结构**
```json
{
  "event": "interview_finished",
  "interview_id": "xxx",
  "user_id": "xxx",
  "finished_at": "2024-01-01T11:00:00Z"
}
```

## 数据流转总览

### 面试主链路延迟目标：3-5秒

```
用户说话（语音）
    ↓ WebRTC
ASR Node 流式识别（~1s，边说边识别）
    ↓
Router Node 状态判断（<50ms，纯代码逻辑）
    ↓
Interview Agent → LLM Node 流式输出（~2-3s）
    ↓ 第一句话输出时立即触发
TTS Node 流式合成（~1s，与LLM并行）
    ↓ SSE
前端流式播放
```

### SFT 数据生产链路

```
面试结束
    ↓
用户填写问卷（对每轮对话打 good/bad 标签）
    ↓ 提交
PostgreSQL 存储标注结果
    ↓ 定期 job
筛选 quality=good 的对话
    ↓
生成 JSONL 文件
    ↓
S3 归档
```

**JSONL 格式**
```json
{
  "messages": [
    {"role": "system", "content": "你是一个技术面试官..."},
    {"role": "assistant", "content": "请介绍一下HashMap的扩容机制"},
    {"role": "user", "content": "用户的回答..."},
    {"role": "assistant", "content": "你提到了负载因子，那如果调成0.5会有什么影响？"}
  ],
  "quality": "good",
  "user_feedback": "追问很有启发"
}
```
